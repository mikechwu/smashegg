# Research Methodology

Conventions for platform-docs and game-rules research in this repo. Adapted from the research discipline of an internal quantitative-research project (the full extraction with file-level provenance lives in local notes, not in this repo); its trading-domain machinery was consciously not adopted (list at the end).

## Adopted practices

1. **Research over memory.** Every claim about a Cloudflare API behavior, a library, or a Guandan rule cites a primary source (URL) with a fetch date — never model memory alone. Time-sensitive claims (pricing, limits, package versions) are the least trustworthy from memory and the first to re-verify.
2. **Multi-source corroboration for ambiguous rules.** A disputed rule needs ≥2 independent sources before it is treated as settled; low-trust sources (unofficial wikis, AI-generated summaries, garbled platform help pages) are explicitly flagged and discarded as uncorroborated. Where sources genuinely conflict, the conflict itself is the finding — documented per source and turned into a RuleVariant config key, not silently resolved.
3. **Per-claim verification tags.** VERIFIED (primary source read) / UNCERTAIN (couldn't confirm) / inference (stated as such) — per claim, not per document. Confidence is graded claim-by-claim.
4. **Null results are first-class, headlined outcomes.** "No source documents X" or "the official text is silent on Y" are findings, recorded in STATUS.md with the same prominence as positive results — and with a diagnosis of *why* the result is null (not indexed? genuinely unspecified? contradictory?). A null accepted without that diagnosis is a bug suspect, not a finding.
5. **Question-first research passes.** Before researching an ambiguous point, write the question, the sources to check, and what evidence would count as "resolved" vs "still ambiguous" — then look. (Implemented as structured research-agent prompts with per-question deliverable formats; prevents motivated source selection.)
6. **Anchoring-free cross-checks.** A verification pass (second model, second researcher) receives the artifact and the question — never the first pass's conclusion. Cross-model lineage diversity (Codex / Gemini / Grok vs Claude-family) is used for load-bearing checks; same-family self-review alone is never sufficient for one.
7. **Audit coverage is documented, not just findings.** Review reports list "checked, no finding" sections alongside defects, so the absence of a finding is distinguishable from the absence of a check.
8. **Named gates with pre-declared acceptance criteria.** Empirical unknowns become named gate checks (G-COMPOSE, G-ALARM, G-WSMETER in PLAN.md §9) with pass conditions written before the experiment runs.
9. **Dated, supersession-marked research docs.** Research files are dated and corrections are appended with explicit "superseded by" markers (e.g. the partysocket version note in cloudflare-facts.md §9) rather than silently rewritten — later readers can see what was known when.
10. **Self-correction logging.** When an earlier framing turns out wrong (including our own review findings against our own docs), the correction is logged in STATUS.md as a process entry, not silently patched.

## Tool & model ladders (current)

Web research runs on the built-in WebSearch/WebFetch tools, `curl` via shell for direct page/PDF fetches, and `gh api` for GitHub content. **Firecrawl is disabled as of 2026-07-13 (credit limit reached — owner instruction); state this in every research-agent prompt.** Escalate to costlier tooling only on demonstrated failure of a cheaper rung, and log the failure.

**External audit panel — headless recipes (settled 2026-07-14, superseding the Q3/TTL round's "needs a TTY" null).** Both CLIs run cleanly from the non-interactive harness; the earlier failures were invocation errors, not platform constraints, and the correct diagnosis took ~10 minutes of reading `--help`:
- **Grok:** `grok --prompt-file <brief.md> --output-format plain --always-approve --no-memory --disable-web-search --max-turns 100 --cwd <scratch-clone>` → clean report on stdout. The failed `grok -p --output-format json` invocations died because `-p/--single` takes the prompt as its VALUE (it swallowed `--output-format` as the prompt); the `nohup grok` TTY error was the interactive TUI — simply the wrong entry point. No PTY tricks needed.
- **Codex:** `codex exec -C <scratch-clone> -s workspace-write -o <report.md> - < <brief.md>` → the final message lands verbatim in the `-o` file (this flag is the "clean flushed report" the earlier rounds lacked; the M4-era EPERM class disappears with `workspace-write` in a scratch clone).
- **Protocol:** run each auditor in its OWN throwaway clone (`git clone <repo> …` + `npm ci`) so an audit agent can never touch the working tree; give both the same brief with a MANDATORY report format ending in a literal `AUDIT VERDICT: <N> findings` line; require a checked-clean list (practice 7). Cost: minutes each, so the panel now runs per-change, not per-milestone.
- **Lesson (practice 4 applied to tooling):** "the tool wants a TTY" is a symptom, not a diagnosed constraint — a tooling null result needs the same *why* as a research null before it is recorded. The round this was settled, the restored panel immediately found a HIGH (the immortal-lobby TTL re-arm) that both the in-house review and 682 green tests had missed.

**Model ladder (owner policy, 2026-07-14 — same escalation rule as the tool ladder).** Trivial → Haiku-class; standard (routine implementation, most UI/polish) → Sonnet-class; hard (architecture, protocol/state-space reasoning, root-cause diagnosis, tricky engine logic) → Opus-class, the default top tier. **Fable is escalation-only**: never routed to automatically, reached solely on demonstrated failure or clear inadequacy of a cheaper rung, with the escalation and its one-line reason logged in STATUS (never escalate preemptively "to be safe"; when in doubt, prefer Opus and note it — Opus proving insufficient and then escalating is the ladder working, not a failure). Per-subtask discipline unchanged: one line stating tier, model, why. Mechanism + verified sources: docs/research/model-dispatch.md. The cross-model audit panel is orthogonal (lineage diversity, not budget) and keeps its own rules.

**Codex allocation — producer ≠ auditor (owner policy, 2026-07-15). A PANEL rule, not a budget rule.** Codex is strong and should carry hard problems and coding tasks — but whoever PRODUCES a change cannot be its independent auditor (practice 6): the verification pass receives the artifact and the question, never the first pass's conclusion, and consultation anchors just as authorship does. If Codex writes OR designs a change, Codex auditing it is same-lineage self-review and the panel silently halves to Grok alone. Our own evidence that two lineages is the *minimum*, not a luxury: Grok called `/purge` clean where Codex flagged it, and Codex missed the TTL re-arm HIGH that Grok caught. Allocation by what the change touches:
- **Load-bearing** (engine, protocol, redaction, timing, the DO's alarm/deadline machinery): Codex stays an AUDITOR — do not let it write or design there. Design + implement in-house (Claude-family), then both lineages audit.
- **Not load-bearing** (UI, presentation, animation, tooling, scripts, test scaffolding): Codex may implement freely — the panel is light there anyway and it is a subscribed resource otherwise idle. Grok + the eyes verify what Codex builds; Codex does not audit its own output.
- **Hard problems:** consulting Codex is fine, but LOG it — it anchors Codex for that change's audit. If the change turns out load-bearing, Grok audits alone and you SAY SO plainly rather than presenting a half panel as a full one.

**True-390 visual verification + the state-driver (settled 2026-07-15, pre-M5 UX round).** Two recipes that make "verify the phone UI" honest:
- **True 390px, despite Chrome's window clamp.** `resize_window` (and the OS) refuses to shrink a Chrome window below ~innerWidth 606, so a "390px" resize silently renders at ~606 — where `vw`-based sizes and a `min-width:720` breakpoint differ from a real phone. Recipe: inject a **same-origin iframe at `width:390px`** pointing at `location.href`; `vw` inside it computes against 390 and shared `localStorage` means it renders the SAME session. Verify `iframe.contentWindow.innerWidth === 390` and `scrollWidth === clientWidth` (no horizontal overflow), then screenshot/zoom the iframe region. **Caveat:** the iframe presents the same seat tokens → a soft socket-takeover disrupts the parent's LIVE session, so use it for read-only capture and reload the parent afterward.
- **State driver for the not-easily-reached states.** A hints-only Node ws bot (`scratchpad/drive.mjs` pattern) creates an untimed room, claims 4 seats, and fast-forwards to any target (tribute / low-card / match-end) by replying to each turn with an action from the server's own `hints` — no engine logic, not clock-limited. It writes `{code, tokens}`; inject those into the browser's `localStorage['room:CODE']` and navigate to render the exact state. This is how tribute / low-card divergence / match-end / an engineered wild-chooser were re-verified in the ring.
- **Flag closed (the record, not the product):** M3/M4's "390px verified" claims were almost certainly made at the clamped ~606, not true 390 — the same class as the e2e-title / phantom-TTL / "no surviving defect" drifts. The substance survives (this round confirms the 27-card hand IS legible at true 390 via the iframe), but henceforth verify at true 390 or state the actual width; never claim "390" loosely.

**QA ratchet (standing rule, added 2026-07-14).** Every computer-use/visual-iteration finding becomes an automated regression (e2e or unit) BEFORE its fix is considered done. QA is a ratchet: manual/visual tooling (browser drives, screenshots, computer use) is for *finding* behavior, never for *keeping* it — whatever a costlier rung of the ladder discovers must be pinned by a cheaper, repeatable rung (a named test that traverses the real product path, e.g. tests/e2e/product-paths.e2e.test.ts) or the finding is considered unfixed. A green test's TITLE must claim only what that green run actually guarantees (e.g. "engine-guaranteed, wire-verified when seed-huntable" rather than a bare wire-level claim for a probabilistic rare path); any stronger proof level a run happens to achieve beyond the titled guarantee must be machine-recorded (not just console-printed) so it is auditable after the fact, and an opt-in strict mode may upgrade "happened to achieve" into "required to pass" for a scheduled/extended run.

## Consciously NOT adopted (trading-domain-specific)

Backtesting engine conventions; walk-forward / purged & embargoed K-fold / CPCV cross-validation; DSR/PBO multiple-testing corrections; market-data hygiene (survivorship bias, adjustment factors, vendor feed semantics); conformal prediction gating; R-multiple/expectancy/SQN reporting; cost/slippage modeling. These guard against temporal leakage and P&L-metric overfitting, which have no analogue in platform-docs/rules research. The *general* principles behind two of them do transfer and are covered above: "distrust reused code, verify before reuse" (→ practice 1) and "count every trial including failures" (→ practice 4).
